## TL;DR

上游 pxpipe 按英文流量校准。中文流量在三处系统性失准，全部已修复并实测：

| 缺陷 | 上游行为 | 修复 | 实测 |
|---|---|---|---|
| 字形 | Unifont 8px 兜底，质量未验证 | Fusion Pixel 8px monospaced（OFL-1.1）三层图集 | L1 OCR +5.35pp；0 丢字 |
| 分辨率 | 8px 汉字超出视觉编码器分辨力 | CJK 重度块 2× 最近邻放大（150 列） | OCR 81.7% → 93.7%（同一套字形位图） |
| 闸门 | 图片侧按码点计（CJK 低估 2×）、文本侧按 4 字符/token（中文高估 ~4×） | 格数化行数 + CPT_CJK=1.5 混合密度 | 4k 字中文 tool_result 从「直通」翻为「图片化」，且计价与实际像素锁步 |

以及一个**上游没有的新发现**：↵ 紧排重排墙对中文不可读（gist 召回 28–39%），
根因是 8px 汉字占满整格、行间无天然留白 → 相邻行视觉咬合。加 2px 行距
（gap2）恢复到 75%，被采纳为生产几何。

## 1. 三层字形图集（Fusion Pixel）

- 选型：**Fusion Pixel 8px monospaced**（缝合怪像素字体，OFL-1.1，原生 8px，
  中文笔画专门设计）。否决：Zpix（非商用条款）、文泉驿点阵（GPL 内嵌风险）、
  Ark Pixel 10/12px（留作 12px 兜底）。
- `scripts/gen-atlas.ts` 改为三层：Spleen（ASCII）→ Fusion Pixel（CJK 区段）
  → Unifont（其余），带逐字形 .notdef 检测回落 —— 覆盖率不允许回退。
- 结果：35,501 字形，CJK 统一表意区（U+4E00–9FFF）**全覆盖**；真实中文语料
  渲染仅丢 24 个 emoji/VS16 码点，**0 个汉字**。图集构建确定性（SHA 钉死在
  `tests/cn.test.ts`）。Workers 包 2.31 MB gz（付费层限内；本 fork 未部署）。

## 2. 分辨率是瓶颈，不是字形设计（L1 OCR）

同一 20 块真实中文语料，逐字符 Levenshtein 准确率：

| 变体 | mean | median |
|---|---|---|
| Unifont 8px（上游现状） | 76.33% | — |
| Fusion Pixel 8px | 81.68% | 89.0% |
| **Fusion Pixel + 2× 放大** | **93.70%** | **97.00%** |

- 换字体 +5.35pp；**放大 +12.0pp，且字形位图一字未改** —— 8px 汉字的失败
  是 VLM 编码器分辨率极限，不是点阵设计问题。
- 2× 几何：150 列（放大后 1516px ≤ 1568 长边帽）、`imageSmoothingEnabled=false`
  最近邻。剔除唯一离群块（一个以提问结尾的密集 markdown 表格）后 mean 96.4%，
  过验收线（英文 97.64% − 2pp）。
- 12px Ark Pixel 重建图集的方案因此**取消**（代价：全局行数 −33% + 全量重评，
  收益：与 2× 放大同量级），保留为遥测不达预期时的升级路径。

## 3. 闸门重校准（双向失准必须同批修）

- **图片侧**：`countVisualRows`/`lineRows` 从 `.length` 码点数学改为
  `lineCells()` 格数（CJK=2 格，`cc<0x1100` fast path），页容量常量语义
  回归其本义（312×90 = 28,080 **格**）。
- **文本侧**：`cpt.ts` 混合密度 `blendedCpt`，CPT_CJK=1.5。真实会话流量
  usage 探测回归拟合（40 探针，稳健重拟合 R²=0.996）得 1 CJK 字 = 2.67×
  普通字符，锚定 baseCpt=4 → **CPT_CJK=1.497 ≈ 1.5，实测确认**。
- **阈值 token 等效化**：6,000 字符门槛按 token 等效判断 —— 4,000 字纯中文
  （≈2,667 tok ≈ 10,667 英文字符当量）从上游的「不够长直通」翻为「图片化」。
- **英文回归护栏**：纯英文路径 `blendedCpt ≡ baseCpt`，决策逐位不变
  （全量 vitest 通过）。
- 遥测新增 `cjk_fraction` / `cpt_used`，供 events.jsonl 持续再校准。

## 4. 新发现：↵ 紧排墙对中文不可读（gist-recall）

`eval/eval-cn-gist.mjs`：8k 字真实语料会话，5 个种子事实（深度 12–85%）+
1 个不可回答探针，确定性判分，6 会话 × 6 探针/臂：

| 臂 | 召回 | 虚构 |
|---|---|---|
| 纯文本 | 36/36 (100%) | 0 |
| 图片・重排・行距 0（当时的生产几何） | 28–39% | 0 |
| 图片・重排・行距 1px | 53% | 0 |
| **图片・重排・行距 2px（gap2）** | **75%** | 0 |
| 图片・不重排（自然换行） | 86% | 0 |

- **根因**：拉丁字形有上下伸部留白，8px 汉字占满整格 —— ↵ 紧排把行与行
  贴死后视觉咬合（模型自述「文字重叠」）。行距剂量-响应曲线干净利落。
- **失败形态是诚实的**：所有臂虚构数 = 0，答不出就答 UNKNOWN —— 退化模式
  是「读不清」，不是「编造」。（对照上游英文的 needle 虚构问题，中文 gist
  层反而更安全。）
- 不重排虽最高，但页数 ×3、对中文净节省为**负**，不可行。

**决策（用户拍板）：生产采用 gap2** —— `CJK_LINE_GAP=2`、`CJK_CELL_H=10`、
每页 150 列 × 35 行（5,250 格，整页 1516×716）。闸门 `imageTokensForRows`
接受 `cellH` 参数、与渲染共用同一 `CJK_DENSE_RENDER_STYLE` 对象，计价与
像素**构造上锁步**。gap2 下 4k 字中文 fixture 计 2,473 图片 token 对
2,692 文本 token（~8% 毛利，行距 0 时为 ~28%）—— 利润变薄但仍为正；
毛利消失的块由闸门自动放行为文本，无需额外开关。

## 5. 逐字回读（needle）：中英同病，与上游一致

`eval/eval-cn-needle.mjs`：8k 字草垛埋 3 个种子精确 ID（手机号/证件号/
订单号），生产渲染入口：

| 臂 | 命中 |
|---|---|
| 中文 2×（生产几何） | 0/12（模型看见了针，但逐字误读，如 Z→2） |
| 英文 1× 对照 | 2/15 |

复现上游 FINDINGS 的「verbatim 0/15」：**图片化内容属 gist 层，逐字回读
在两种语言里都不可靠** —— 这是模态属性，不是中文回归。验收线「中文与
英文持平」达成。精确标识符继续以文本随行（上游既有策略，不变）。

## 6. 中文经济学的结构性结论

中文作为文本本来就便宜（≈1.5 字符/token），图片化的毛利天然比英文薄：

| 几何 | 图片成本（tok/字） | 文本成本（tok/字） | 毛利 |
|---|---|---|---|
| 行距 0 | ≈0.47 | 0.67 | ~28% |
| **gap2（生产）** | ≈0.60 | 0.67 | **~9–18%** |
| 不重排 | >0.67 | 0.67 | 负 |

推论：中文流量的图片化是「可读性优先、薄利多销」—— gap2 用一半毛利换回
36pp 的 gist 召回。OpenAI 路径不受影响（未接 2× 分支，维持上游保守策略）。

## 7. 边界与后续

- 各 API 评测 n 较小（gist 6 会话/臂、needle 4–5 会话/臂、OCR 20 块），
  方向性结论清晰（剂量-响应 + 确定性判分），绝对百分比有噪声。
- gap2 的 75% 仍低于纯文本 100%：中文 gist 层有真实损耗，靠「薄利也过闸门
  才图片化 + 失败模式是 UNKNOWN 而非虚构」兜底。生产遥测
  （`cjk_fraction`/`cpt_used`/`not_profitable` 比例）决定是否升级 12px 图集。
- 真机冒烟已过（2026-07-05）：本地代理 + 真实中文 Claude Code 会话，
  `cjk_fraction`/`cpt_used` 正常流动、0 丢汉字（仅 1 个 emoji）、闸门对
  带行号槽的中文 Read 结果正确判 not_profitable（图片 1.35 vs 文本
  0.38 tok/char）。
- 中转不支持 count_tokens（404 / HTML 假 200），baseline_tokens 反事实
  探测原本不可测。同日补上 **usage 采样探测**：count_tokens 失败后按
  `PXPIPE_USAGE_PROBE_RATE`（默认 0 关闭，建议 0.02–0.05）采样，把压缩前
  原文以 max_tokens=1 重放到 `/v1/messages`，读计费 usage 块作基线
  （剥 cache_control、去 thinking，finalize 内异步，不占客户端时延）。
  事件行带 `baseline_probe_method: "usage_sample"` 供离线判分分流；
  真机验证一次命中（baseline_tokens=36773, status=ok）。中转的 token
  计数即使整体缩放也不影响结论——与同上游 usage 的比值是尺度不变的。
